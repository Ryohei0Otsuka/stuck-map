# Stuck Map

**Stuck Map** は、プロジェクト進行中の小さな詰まり・確認待ち・認識合わせ・レビュー依頼を、軽いサインとして置ける進行支援ボードです。

> 誰が遅いかではなく、何が詰まっているかを見る。

タスクの進捗だけではなく、作業・確認・認識合わせ・レビュー・HELP の流れがどこで止まりそうかを早めに見つけることを目的にしています。

## 開発背景

プロジェクトでは、タスクそのものの難しさだけでなく、確認待ち・判断待ち・レビュー待ち・相談しづらさによって進行が止まることがあります。

たとえば、

- 忙しい上司や先輩に声をかけづらい
- 確認したいが、今聞いてよいか迷う
- 何待ちなのか分からないまま作業が止まる
- 既読はつくが、その後の状態が分からない
- 判断や確認が特定の人に集中する
- 電話・口頭・チャットに情報が散らばり、状態が残りにくい

こうした詰まりは、必ずしも誰か一人の努力不足で起きるものではありません。

「分からないなら聞けばいい」  
「早めに共有すればいい」  
「タスク管理ツールに書けばいい」

それらは正論です。

ただ実際には、その前に「聞くか迷う」「誰に聞くべきか分からない」「課題化するほどか分からない」「相手に負荷をかけたくない」といった心理的・構造的な段差があります。

Stuck Map は、そうした小さな詰まりを、重い報告ではなく軽いサインとして置けるようにするためのプロトタイプです。

---

## なぜ作ったか

業務の中で、情報が止まると、人も止まります。

確認待ちや判断待ちが見えないと、作業者は、

- 催促してよいのか
- 次に進んでよいのか
- 自分が何か間違えたのか
- 返答を待つべきなのか

を考え続けることになります。

これは、見えにくい認知コストです。

プロジェクトが止まるとき、原因は必ずしも個人の能力不足や努力不足ではありません。

情報の流れが止まる。  
確認が一部の人に集中する。  
判断基準が共有されていない。  
相談の入口が見えない。  
小さな違和感が、正式な課題になる前に埋もれる。

そうした状態は、プロジェクトの流れに起きる小さな不調です。

放置すれば、確認待ち・判断待ち・属人化・遠慮として積み重なり、やがて大きな停滞になります。

Stuck Map は、そうした詰まりを「根性」や「もっと自発的に動くべき」という言葉だけで片づけず、構造として扱うための試みです。

個人に自発性を求めるだけではなく、その自発が届き、受け止められ、流れに戻れる環境を整える。

それが、このプロトタイプの出発点です。

## Concept

Stuck Map の中心思想は、次の一文です。

> 誰が遅いかではなく、何が詰まっているかを見る。

Stuck Map は、個人の遅れや生産性を監視するためのものではありません。

見る対象は、人ではなく、作業・確認・認識合わせ・レビュー・HELP の流れです。

物理的な物流では、荷物がどこにあるかを追跡できます。

一方で、業務上の確認・判断・レビュー・返答は、どこで止まっているのか見えにくいことがあります。

Stuck Map は、その「情報の流れ」や「待ちの状態」に軽いラベルを付けることで、プロジェクトの流れを止めにくくすることを目指しています。

---

## What Stuck Map Shows

Stuck Map では、タスクの状態を大きく2つに分けています。

### 1. フロー

タスクそのものの進行状態です。

| 表示 | 補助コード | 意味 |
|---|---|---|
| これから | TODO | まだ着手していないもの |
| 作業中 | DOING | 現在進めているもの |
| 完了 | DONE | 作業が終わったもの |

### 2. サイン

タスクに付けられる、確認・相談・支援の状態です。

| 表示 | 補助コード | 意味 |
|---|---|---|
| お助け | HELP | 少し手を借りたいもの |
| 確認依頼 | CHECK | 軽く確認してほしいもの |
| 認識合わせ | ALIGN | 方針や前提を合わせたいもの |
| レビュー依頼 | REVIEW | 一度見てほしいもの |

サインは、タスクを完了させるためのボタンではありません。

たとえば、`作業中` のタスクに `お助け` サインが付いている場合、そのタスクはフロー上では作業中のまま残り、サイン側では「お助け」として拾えるようになります。

サインを解消しても、タスク自体は完了にはなりません。  
タスクを完了させる場合は、フロー側または詳細画面で完了操作を行います。

## Design Notes

### フローとサインを分ける

Stuck Map では、タスクの進行状態と、詰まり・確認・相談の状態を分けています。

- **フロー**：タスクそのものの進行
- **サイン**：詰まり・確認・認識合わせ・レビュー依頼

サインを出しても、タスクはフロー上に残ります。

これにより、次のような状態を扱いやすくしています。

- 作業は進行中だが、確認だけ必要
- 完了ではないが、レビュー待ち
- 止まってはいないが、前提を合わせたい
- 誰かが少し拾えば流れる

---

### 自発を求めるだけでなく、受け止める構造を作る

「困ったら早めに言ってね」という言葉は大切です。

しかし、それだけでは足りない場面があります。

相談する側には、

- 今聞いてよいのか
- まだ自分で考えるべきなのか
- 誰に聞けばよいのか
- こんな小さいことで止めてよいのか
- 相手の負荷を増やしてしまわないか

という迷いがあります。

一方で、受け止める側にも、

- どこで誰が止まりそうか見えない
- 何を優先して拾うべきか分からない
- 確認や判断が自分に集まりすぎている
- 同じ種類の相談が繰り返されている

という見えにくい負荷があります。

Stuck Map は、相談する側の自発性だけに頼るのではなく、そのサインをチームが受け止めやすくするための構造を作ることを目指しています。

---

### HELPを増やすだけでなく、HELPの流れを整える

Stuck Map は、単に「助けて」と言いやすくするだけのツールではありません。

確認や相談が特定の人に集まり続ける場合、それは個人の問題ではなく、チームの流れや役割分担、手順の共有不足が原因かもしれません。

Stuck Map では、こうした状態を責める表示にはせず、次のようなことを考える材料として扱います。

- 同じ人に確認が集まりすぎていないか
- 上司やリーダーでなくても拾える確認が混ざっていないか
- 手順書化・ナレッジ化できるものがないか
- 判断基準を共有できないか

目的は、HELPを増やすことではありません。

**HELPの流れを整え、特定の人に集中しすぎないようにすること** です。

---

### 属人化の兆候を見つける

同じ人に確認・判断・レビューが集まり続ける場合、それは個人の問題ではなく、手順や判断基準が共有されていない属人化の兆候かもしれません。

たとえば、

- その人しか知っていない手順がある
- 判断基準が共有されていない
- 過去の経緯が一部の人にしか残っていない
- 誰に聞けばよいか分からない
- 同じ種類の確認が繰り返されている

といった状態です。

Stuck Map は、HELP / CHECK / ALIGN / REVIEW などのサインがどこに集まりやすいかを見ることで、属人化している作業や判断に気づくきっかけを作ります。

目的は責任追及ではありません。

繰り返し発生する確認や相談を見つけ、

- 手順書化できないか
- 判断基準を共有できないか
- 他の人でも拾える形にできないか
- ナレッジとして残せないか

を考えるための材料にすることです。

Stuck Map は、属人化を一気に解消するものではありません。

ただし、属人化している場所を「見えないまま」にせず、チームで扱える形にするための入口になることを目指しています。

---

### 既読後の空白を減らす

業務では、確認依頼を出した後に「既読はついたが、その後の状態が分からない」ということがあります。

未読なら「まだ見ていない」と分かります。  
返答があれば「OK」「修正」「保留」が分かります。

しかし、既読だけでは、

- 確認中なのか
- 後で返すつもりなのか
- 問題なしなのか
- 判断待ちなのか
- 上位確認中なのか

が分かりません。

Stuck Map は、そうした待ちの宙ぶらりん感を、軽いサインとして扱えるようにすることを意識しています。

---

### 循環を見る

Stuck Map が見たいのは、個々のタスクだけではありません。

プロジェクトの中で、

- 作業が進む
- 確認が出る
- 誰かが拾う
- 判断が返る
- レビューされる
- また作業に戻る

という循環が、どこで弱くなっているかです。

詰まりは、プロジェクトの循環に起きる小さな不調です。

早く見つけて、軽く手当てできれば、大きな停滞になる前に流れへ戻せます。

Stuck Map は、そのための地図です。

---

### 進捗会議の前に、小さな詰まりを拾う

Stuck Map は、問題が大きくなってから報告するためのものではありません。

日々の作業の中で、

- 少し確認したいこと
- 方向を合わせたいこと
- 誰かに見てほしいこと
- 判断待ちになっていること
- 一人では進みにくくなっていること

を軽く置き、チームが早めに拾える状態を作ることを目指しています。

進捗会議で初めて問題が見えるのではなく、普段の流れの中で小さな詰まりを共有できること。

それが、このプロトタイプで大切にしたことです。

## Features

- プロジェクト名・メモの編集
- メンバー管理
- タスクの追加・編集・削除
- タスクへのサイン追加

- localStorage による自動保存
- 初回説明モーダル

## Data Storage

現在の Stuck Map は、ブラウザの `localStorage` にデータを保存します。

そのため、以下のような用途に向いています。

- ポートフォリオ用のデモ
- 個人での試用
- プロトタイプ検証
- UI / UX の確認
- チーム進行支援アイデアの共有

一方で、現時点では以下には対応していません。

- ログイン
- 複数端末同期
- チーム共有
- リアルタイム同期
- 権限管理
- 履歴保存
- サーバーDB保存

実運用サービスとして使う場合は、認証、DB、権限管理、バックアップなどの追加設計が必要です。

## Positioning

Stuck Map は、既存のタスク管理や課題管理の仕組みを置き換えるものではありません。

それらは、正式な課題やタスクを管理する上で非常に強力です。

Stuck Map が扱いたいのは、その前段階です。

正式な課題として登録するほど整理されていないけれど、

- 少し聞きづらい
- 確認したい
- 認識を合わせたい
- レビューしてほしい
- 誰に聞けばよいか分からない
- 何待ちなのか分からない
- HELPを出すには少し重い

といった小さな詰まりを、軽いサインとして置けるようにすることを目的としています。

既存のタスク管理を置き換えるのではなく、そこに載る前の違和感や確認待ちを拾うための補助ボードです。

---

## Prototype Scope

現在の Stuck Map は、チーム共有サービスとして完成しているものではありません。

現時点では、ブラウザ内に保存されるプロトタイプであり、主に以下の用途を想定しています。

- 個人の詰まり整理
- 作業前後の状態確認
- 朝会・定例・引き継ぎでの会話材料
- ポートフォリオ用デモ
- チーム進行支援アイデアの検証

特に現在の localStorage 版は、まず「個人の作戦ノート」としての位置づけです。

将来的には、複数人が同じ盤面を見ながら、詰まり・確認・レビュー・HELP を置き、拾い、流れに戻せるような、共同編集型の進行支援ボードを目指しています。

## Current Focus

現在は、機能追加よりも UI 整理と挙動の安定化を優先しています。

特に、

- 進捗とサインを混ぜない
- タスク追加を軽くする
- サイン追加時の導線を分かりやすくする
- 右パネルで「次に拾うサイン」を分かりやすくする
- 画面を見て3秒で「今どこが詰まっているか」が分かる状態に近づける
- 説明文を増やしすぎず、使う画面として整える

ことを重視しています。
