# Stuck Map

**Stuck Map** は、プロジェクトの中で発生する「確認待ち」「判断待ち」「少し手を借りたいこと」などの小さな詰まりを、軽いサインとして見える化する進行支援ボードです。

> 誰が遅いかではなく、何が詰まっているかを見る。

タスクの進捗だけではなく、作業・確認・判断・レビュー・HELP の流れがどこで止まりそうかを早めに見つけることを目的にしています。

## 背景

プロジェクトでは、タスクそのものの難しさだけでなく、確認待ち・判断待ち・レビュー待ち・相談しづらさによって進行が止まることがあります。

たとえば、

- 忙しい上司や先輩に声をかけづらい
- 確認したいが、今聞いてよいか迷う
- 何待ちなのか分からないまま作業が止まる
- 既読はつくが、その後の状態が分からない
- 判断や確認が特定の人に集中する
- 電話・口頭・チャットに情報が散らばり、状態が残りにくい

こうした詰まりは、必ずしも誰か一人の努力不足で起きるものではありません。

「分からないなら聞けばいい」  
「早めに共有すればいい」  
「タスク管理ツールに書けばいい」

それらは正論です。

ただ実際には、その前に「聞くか迷う」「誰に聞くべきか分からない」「課題化するほどか分からない」「相手に負荷をかけたくない」といった心理的・構造的な段差があります。

Stuck Map は、そうした小さな詰まりを、重い報告ではなく軽いサインとして置けるようにするためのプロトタイプです。

---

## コンセプト

Stuck Map の中心思想は、次の一文です。

> 誰が遅いかではなく、何が詰まっているかを見る。

Stuck Map は、個人の遅れや生産性を監視するためのものではありません。

見る対象は、人ではなく、作業・確認・判断・レビュー・HELP の流れです。

物理的な物流では、荷物がどこにあるかを追跡できます。

一方で、業務上の確認・判断・レビュー・返答は、どこで止まっているのか見えにくいことがあります。

Stuck Map は、その「情報の流れ」や「待ちの状態」に軽いラベルを付けることで、プロジェクトの流れを止めにくくすることを目指しています。

---

## Stuck Map で見えるもの

Stuck Map では、タスクの状態を大きく2つに分けています。

### 1. フロー

タスクそのものの進行状態です。

| 表示 | 補助コード | 意味 |
|---|---|---|
| これから | TODO | まだ着手していないもの |
| 作業中 | DOING | 現在進めているもの |
| 完了 | DONE | 作業が終わったもの |

### 2. サイン

タスクに付けられる、確認・相談・支援の状態です。

| 表示 | 補助コード | 意味 |
|---|---|---|
| 手助け | HELP | 少し手を借りたいもの |
| 方向相談 | WAIT | 方針や判断を合わせたいもの |
| 確認 | CHECK | 軽く確認してほしいもの |
| レビュー | REVIEW | 一度見てほしいもの |

サインは、タスクを完了させるためのボタンではありません。

たとえば、`作業中` のタスクに `手助け` サインが付いている場合、そのタスクはフロー上では作業中のまま残り、サイン側では「手助け」として拾えるようになります。

サインを解消しても、タスク自体は完了にはなりません。  
タスクを完了させる場合は、フロー側または詳細画面で完了操作を行います。

## このプロトタイプで大切にしたこと

### 1. 相談のハードルを下げる

「助けてください」と大きく言う前に、軽く **手助け** のサインを置けることを大切にしています。

実務では、質問や相談の内容が小さいほど、かえって声をかけるタイミングに迷うことがあります。

忙しそうな人に話しかけづらい。  
まだ自分で考えるべきか迷う。  
こんなことで止めていいのか不安になる。

そうした小さな迷いを、軽いサインとして置けるようにすることで、詰まりが大きくなる前に拾える状態を目指しました。

---

### 2. 局所集中を見えるようにする

確認や判断が同じ人に集まると、その人が悪いわけではなくても、プロジェクト全体の流れが止まりやすくなります。

Stuck Map では、そうした状態を個人の負荷として責めるのではなく、チームの流れや役割分担のサインとして見ます。

たとえば、

- 同じ人に確認が集まりすぎていないか
- 同じ種類の判断が繰り返されていないか
- 上司でなくても拾える確認が混ざっていないか
- 手順書化・ナレッジ化できるものがないか

を見るきっかけにします。

目的は、HELPを増やすことではありません。

**HELPの流れを整え、特定の人に集中しすぎないようにすること** です。

---

### 3. 属人化の兆候を見つける

同じ人に確認・判断・レビューが集まり続ける場合、それは個人の問題ではなく、手順や判断基準が共有されていない属人化の兆候かもしれません。

たとえば、

- その人しか知っていない手順がある
- 判断基準が共有されていない
- 過去の経緯が一部の人にしか残っていない
- 誰に聞けばよいか分からない
- 同じ種類の確認が繰り返されている

といった状態です。

Stuck Map は、HELP / CHECK / WAIT / REVIEW などのサインがどこに集まりやすいかを見ることで、属人化している作業や判断に気づくきっかけを作ります。

目的は責任追及ではありません。

繰り返し発生する確認や相談を見つけ、

- 手順書化できないか
- 判断基準を共有できないか
- 他の人でも拾える形にできないか
- ナレッジとして残せないか

を考えるための材料にすることです。

Stuck Map は、属人化を一気に解消するものではありません。

ただし、属人化している場所を「見えないまま」にせず、チームで扱える形にするための入口になることを目指しています。

---

### 4. 既読後の空白を減らす

業務では、確認依頼を出した後に「既読はついたが、その後の状態が分からない」ということがあります。

未読なら「まだ見ていない」と分かります。  
返答があれば「OK」「修正」「保留」が分かります。

しかし、既読だけでは、

- 確認中なのか
- 後で返すつもりなのか
- 問題なしなのか
- 判断待ちなのか
- 上位確認中なのか

が分かりません。

Stuck Map では、こうした状態を `WAIT` や `CHECK` などのサインとして置くことで、待ちの宙ぶらりん感を少しでも減らすことを意識しています。

---

### 5. 人のリソースを守る

情報が止まると、人も止まります。

確認待ちや判断待ちが見えないと、作業者は、

- 催促してよいのか
- 次に進んでよいのか
- 自分が何か間違えたのか
- 返答を待つべきなのか

を考え続けることになります。

これは、見えにくい認知コストです。

Stuck Map は、そうした見えない待ちを軽いサインとして置くことで、人の集中力や時間が余計に削られない状態を目指しています。

---

### 6. 進捗会議の前に、小さな詰まりを拾う

Stuck Map は、問題が大きくなってから報告するためのものではありません。

日々の作業の中で、

- 少し確認したいこと
- 方向を合わせたいこと
- 誰かに見てほしいこと
- 判断待ちになっていること
- 一人では進みにくくなっていること

を軽く置き、チームが早めに拾える状態を作ることを目指しています。

進捗会議で初めて問題が見えるのではなく、普段の流れの中で小さな詰まりを共有できること。

それが、このプロトタイプで大切にしたことです。

## データ保存について

現在の Stuck Map は、ブラウザの `localStorage` にデータを保存します。

そのため、以下のような用途に向いています。

- ポートフォリオ用のデモ
- 個人での試用
- プロトタイプ検証
- UI / UX の確認
- チーム進行支援アイデアの共有

一方で、現時点では以下には対応していません。

- ログイン
- 複数端末同期
- チーム共有
- リアルタイム同期
- 権限管理
- 履歴保存
- サーバーDB保存

実運用サービスとして使う場合は、認証、DB、権限管理、バックアップなどの追加設計が必要です。

## 位置づけ

Stuck Map は、既存のタスク管理や課題管理の仕組みを置き換えるものではありません。

それらは、正式な課題やタスクを管理する上で非常に強力です。

Stuck Map が扱いたいのは、その前段階です。

正式な課題として登録するほど整理されていないけれど、

- 少し聞きづらい
- 確認したい
- 判断待ちになっている
- 誰に聞けばよいか分からない
- 何待ちなのか分からない
- HELPを出すには少し重い

といった小さな詰まりを、軽いサインとして置けるようにすることを目的としています。

既存のタスク管理を置き換えるのではなく、そこに載る前の違和感や確認待ちを拾うための補助ボードです。

## 今後追加したい機能

- 拾ったサインの軽い記録
- 詰まり理由の整理表示

- HELPの局所集中検知

ただし、監視感が強くなる機能は慎重に扱います。

特に、個人別の負荷グラフや生産性スコアのような機能は、Stuck Map の思想とは相性が悪い可能性があります。

---

## 現在の課題

プロトタイプとして機能を増やした結果、画面上の情報量が多くなっています。

今後は、機能追加よりも UI 整理を優先します。

特に、

- 進捗とサインを分ける
- タスク追加を軽くする
- サイン追加時に理由・相手・期待する返答を扱う
- HELPが特定の人に集中しないようにする
- 右パネルで「次に拾うサイン」を分かりやすくする

といった改善を行う予定です。
